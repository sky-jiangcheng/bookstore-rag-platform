# 重构历史

本文档整理的是仓库当前代码中能够观察到的主要演进阶段，用于帮助后续维护者理解为什么会同时存在多套实现、哪些是主线、哪些是兼容层。

## 总体脉络

项目从“基础书店管理”逐步演进到“RAG + 多模型 + Agent 书单生成”的业务系统。每一轮重构都在解决一个明确问题，但也留下了一些兼容层和历史实现，因此当前更需要收口而不是继续分叉。

## 阶段一：基础业务能力

- 先完成图书、用户、采购、导入、补货等基础 CRUD 能力
- 后端以 FastAPI 承载业务路由
- 前端以 Vue 3 + Element Plus 承载管理界面
- 数据层以 MySQL 存储结构化业务数据

这一阶段奠定了项目的业务骨架，但推荐与相似度搜索能力还比较初级。

## 阶段二：RAG 与向量检索重构

- 引入独立向量数据库，把相似度搜索从 MySQL 里拆出来
- 把“直接查询全量书籍”的方式改成“向量检索 + 数据库补充查询”
- 将查重、推荐、补货等能力逐步和检索层绑定

这一阶段的核心目标是提升检索性能和扩展性。当前主线已经以 Qdrant 为准，LanceDB 兼容封装属于历史残留。

## 阶段三：LLM 服务抽象

- 把大模型调用抽象成 provider 结构
- 兼容 OpenAI、Gemini、Qwen、Ernie、Zhipu、Local、Mock 等提供商
- 增加降级策略，避免单一模型不可用导致主流程中断

这一阶段解决的是“模型可替换”和“失败可降级”的问题，但也使配置项变多，需要后续持续收口。

## 阶段四：交互式书单与 Agent 化

- 书单推荐从单次返回演进到需求解析、需求细化、生成书单的多步流程
- 新增 `v2/agent` 接口，支持 HTTP、SSE 和 WebSocket
- 前端新增 Agent 书单助手页面和状态管理

这一阶段显著增强了用户体验，但也引入了新的 API 前缀和前后端对接要求，容易与旧的 `/api/v1` 体系混淆。

## 阶段五：配置统一与启动收口

- `config.yml` 作为主配置源
- 环境变量用于覆盖敏感项
- 增加启动前检查、服务注册、统一入口
- 保留少量兼容脚本和历史入口，方便平滑迁移

这一阶段的目标是把分散的配置和启动逻辑统一起来，降低部署和维护成本。

## 阶段六：文档和入口整理

- 主 README 改成结构化总览
- 重构历史单独拆出
- 核心模块说明收敛成索引式文档

这是当前正在做的收口工作，目标是减少重复说明，让维护者一眼找到主线。

## 当前遗留的兼容层

- `backend/main_fixed.py` 作为历史兼容入口保留
- `backend/app/utils/vector_db.py` 仍保留 LanceDB 兼容实现
- `backend/app/services/gemini_service.py` 仍依赖旧的 Gemini 兼容 SDK
- `backend/tests/TEST_REPORT_REQUIREMENT_AGENT.md` 保留为历史测试记录

这些文件不一定都要立即删除，但需要明确它们的定位是“兼容/历史”而不是“主线实现”。

## 建议的后续收口顺序

1. 继续减少历史入口和兼容分支
2. 把前端大体积 chunk 做 code splitting
3. 把旧 Gemini 兼容 SDK 迁移到新 SDK
4. 统一所有 API 基线和文档示例
